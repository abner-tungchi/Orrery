v2.2.0

解決 AI 多帳號的救星

為 Claude Code、Codex、Gemini CLI 隔離帳號，自由切換工作與個人情境 — 對話歷史無縫接續，不再從頭開始。

開始使用 GitHub
透過 Homebrew 安裝
$ brew install OffskyLab/orrery/orrery
或透過 APT 安裝（Ubuntu / Debian）
$ echo "deb [trusted=yes] https://offskylab.github.io/apt stable main" \
  | sudo tee /etc/apt/sources.list.d/orrery.list
$ sudo apt update && sudo apt install orrery
啟用 Shell 整合
$ source ~/.orrery/activate.sh

無痛切換所需的一切

🔒

自由切換，告別繁瑣登入

設定完成後，一行指令切換工作和個人帳號。每個環境的憑證完全隔離 — 不用登出、不用重新輸入密碼、不用等認證流程。

💬

Session 共享

切換帳號後直接接續對話。Session 資料預設共享 — claude --resume 在切換環境後無縫運作。

🐚

Per-shell 生效

orrery use work 只在當前 shell 設定環境變數。其他終端機視窗維持各自的環境，互不影響。

✦

互動式精靈

透過方向鍵與空白鍵選擇工具，引導式建立環境 — 不需要記任何參數。

🔑

自訂環境變數

每個環境可儲存 API 金鑰等變數，orrery use 切換時自動匯出。

⚡

零執行開銷

Swift 原生編譯，啟動瞬間完成。Shell 整合只是一個輕量的 eval 函式，不是背景程式。支援 bash 與 zsh。

運作方式

1

安裝 Shell 整合

執行一次 orrery setup。會產生 ~/.orrery/activate.sh 並寫入 rc 檔。再執行 source ~/.orrery/activate.sh 即可在當前 shell 生效。

activate.sh 也會在背景每天檢查是否有新版本，並在下次開啟 shell 時顯示通知。

2

建立環境

執行 orrery create work。互動式精靈讓你選擇工具並設定 session 共享。環境存放在 ~/.orrery/envs/。

3

一行指令切換

orrery use work 將正確的設定目錄與自訂環境變數匯出到當前 session。對話歷史在切換後完整保留。

4

接續上次的對話

切換帳號後，claude --resume 直接接續上一個 session。認證隔離，session 共享 — 兩全其美。

origin 環境

不只是另一個環境 — 這是系統的原始狀態

origin 是 Orrery 的保留名稱，代表未經任何修改的系統原始環境。 執行 orrery use origin（或 orrery deactivate）後， Orrery 會清除所有隔離設定，讓工具回到系統全域設定， 就像 Orrery 從未安裝過一樣。

與一般環境不同，origin 無法被刪除、重新命名， 也不支援工具設定或環境變數。它沒有任何隔離、也沒有獨立的 memory 或 session — 它就是進入 Orrery 之前的狀態。

這讓 origin 成為一個乾淨的退出點： 想用系統預設憑證執行指令時，不需要切換帳號，也不會影響任何 Orrery 環境。

應用情境

💼

工作 / 個人帳號

公司用 Claude Team plan，自己有 Pro 訂閱。不用反覆登入登出，一行指令切換。

$ orrery use work
# → 使用公司 Team plan

$ orrery use personal
# → 使用個人 Pro plan
🏢

多客戶 / 多專案

接案者或顧問同時服務多個客戶，各自有不同的 API key 和憑證。一行指令切換情境。

$ orrery create client-a --tool claude
$ orrery create client-b --tool claude
$ orrery use client-a
💬

切換帳號不中斷對話

用工作帳號跟 Claude 對話到一半？切到個人帳號處理事情，再切回來 — session 還在原地等你。

$ orrery use work
$ claude
# ... 工作中 ...

$ orrery use personal
$ orrery use work
$ claude --resume
# → 接續上次的對話

AI 工具整合

Orrery 透過 MCP 整合 Claude Code、Codex CLI 和 Gemini CLI。跨帳號委派任務、工具間共享記憶、管理 session——全在一個地方。

1

設定一次就好

在專案目錄下執行。Orrery 會註冊為 MCP server 並安裝 slash commands。

$ orrery mcp setup
2

Slash commands

設定完成後，在 Claude Code 裡直接使用：

/orrery:delegate 委派任務給其他帳號
/orrery:delegate 用工作帳號 review 這個 PR 的安全問題
/orrery:sessions 列出 session 並用 index 接續
/orrery:sessions
接著： orrery resume 1
3

或直接用自然語言

Claude 透過 MCP tools 認識你的所有環境。你可以這樣說：

→ 「用工作帳號 review 這個 PR」
→ 「我現在是哪個帳號？」
→ 「列出最近的 session，幫我接續上一個」
4

MCP 工具與共享記憶

所有 AI 工具自動取得以下工具。任一工具儲存的知識，其他工具都能存取。

orrery_memory_read 讀取共享專案記憶
orrery_memory_write 寫入共享專案記憶
orrery_delegate 用其他帳號執行任務
orrery_list 列出所有環境
orrery_sessions 列出當前專案的 session
orrery_current 查看目前啟用的環境
5

外部記憶儲存

Memory 預設儲存在 ~/.orrery。你可以將它重新導向到任何目錄，包括 Obsidian 等 wiki 系統，讓 AI 記憶和你的筆記放在一起。

$ orrery memory storage ~/Documents/my-wiki/orrery
新路徑目前沒有記憶，是否要將目前的記憶複製過去？
▶ 複製過去
不複製（從空白開始）

Fragment 機制與 AI 整合方式不變，AI 直接讀寫你的 wiki 資料夾。

P2P 記憶同步 實驗性功能

跨機器、跨團隊即時同步專案記憶，點對點直連。

🌐

桌機 + 筆電

同一人、兩台電腦。在同一網路下啟動 daemon，Bonjour 自動發現彼此。

# 桌機
$ orrery sync daemon --port 9527

# 筆電（Bonjour 自動發現）
$ orrery sync daemon --port 9528
👥

團隊協作

建立團隊、分享邀請碼。隊友加入後啟動 daemon，所有人的記憶即時同步。

$ orrery sync team create my-team
$ orrery sync team invite --port 9527
# → 分享邀請碼

# 隊友加入：
$ orrery sync team join <code>
$ orrery sync daemon --port 9528
🔒

加密傳輸（mTLS）

敏感環境適用。雙方必須出示同一 CA 簽發的憑證，未授權的節點在握手階段即被拒絕。

$ orrery sync daemon --port 9527 \
  --tls-ca ca.pem \
  --tls-cert node.pem \
  --tls-key node-key.pem
🌍

跨網段

不同辦公室、不同網路。輕量的 rendezvous server 負責節點發現，資料直接在節點間傳輸。

# VPS 上：
$ orrery sync rendezvous --port 9600

# 各節點：
$ orrery sync daemon --port 9527 \
  --rendezvous rv.example.com:9600

僅同步專案記憶 — session 留在本機（因為它綁定機器路徑）。新成員首次連線時透過 manifest reconcile 取得所有現有記憶。記憶變更以無衝突的 fragment 追蹤，由 AI agent 在 session 啟動時整合。

版本歷程

版本說明 — 完整技術細節在 CHANGELOG。

v2.0.0 最新

改名為 Orrery

為什麼要改名？因為 orbital 作為軟體名稱太常見了，搜尋起來常常撞到別的工具。這版改成 orrery（原意是太陽系機械模型）— 主題上更貼近這個專案「維護多個平行設定宇宙」的精神，名稱也夠冷門。

內容沒有任何功能差異，純粹就是改名：CLI binary orbital → orrery、設定目錄 ~/.orbital/ → ~/.orrery/、環境變數 ORBITAL_* → ORRERY_*、Swift module OrbitalCore → OrreryCore、Homebrew tap OffskyLab/orrery/orrery。

自動搬家：第一次跑 orrery 時，如果偵測到 ~/.orbital/（而且沒有 ~/.orrery/），會自動把目錄整個搬過來，順便重新產生 activate.sh、更新 rc 檔的 source 路徑。舊的 Orbital repo 保留一版 deprecated wrapper — 你現有的 orbital 指令、MCP 設定、shell alias 都還會正常運作，只是會印 deprecation 提示。

v1.1.6

換帳號不用再放棄已經調好的設定

以前想把一個 env 複製出來、換個帳號用會遇到兩難：重新登入麻煩，但整份複製又會連「哪個帳號」都被覆蓋掉，連帶 theme、plugin、已經看過的提示全都一起變了。這版我們把「要用哪個帳號登入」和「要複製誰的設定」拆成兩個獨立的步驟 — 你可以從 A 拿帳號、從 B 拿偏好，.claude.json 會自動幫你合併：身份從登入來源來，其他偏好從 clone 來源來，per-account 快取清掉讓 Claude 自己重抓。

同時每個 tool 都有自己的 login + clone 流程，建立環境時一個一個問「新增並設定？」，不再一口氣把三個工具綁在一起。orrery tools 也拆成 tools add / tools remove；session 共享/隔離改成 per-tool，「Claude 共享、Codex 隔離」這種組合終於行得通；orrery list 跟 orrery info 也會直接顯示每個 tool 登入的 email 和方案。

v1.1.5

Wizard 選完會自己清乾淨

為什麼要動 wizard 的顯示？因為之前每選完一步，那整塊選單都還留在畫面上，跑完全部步驟後 scrollback 滿滿都是中間過程，真正重要的摘要反而被埋住。這版每步選完自動清掉，只留最終結果；create 完後也會直接問你要不要立刻切過去，省得再手動跑一次 orrery use。順手也拿掉了早就不能用的 claude auth login 步驟 — 它根本不認 CLAUDE_CONFIG_DIR，留著只會誤導。

v1.1.4

版本檢查不再卡 shell 啟動

用 Powerlevel10k instant prompt 的朋友反應 shell 一啟動 prompt 就會被 orrery 的版本檢查打斷，原因是我們在 shell init 階段同步做了這件事。這版改成「你下次跑 orrery 指令時順便在背景查一下」，最多 4 小時一次。發現新版時會用黃色訊息出現在每次 orrery 指令前，直到你執行 orrery update 才會消失。

v1.1.3

orrery use 跨 shell 會記得住

為什麼要修這個？因為以前在一個 terminal 切到 work，打開另一個 terminal 卻發現自己又回到了 origin — 其實是 orrery use 切完之後沒把結果寫進 current pointer，新開的 shell 根本讀不到。這版每次 use 成功都會正確寫入，新開的 shell 就會自動回到你上次使用的環境。

v1.1.2

Memory 不用再在 CLAUDE.md 手動寫指示

以前想讓 Claude 讀 orrery 的共享記憶，你得自己去 CLAUDE.md 裡加一段「請去某某路徑讀」的指示，Claude 才會理你 — 很違反「預設就 work」的精神。這版每個專案的 Claude auto-memory 資料夾會自動 symlink 到 orrery 的共享記憶，什麼設定都不用寫，Claude 自己就會讀到；跨 env、跨機器都是同一份。

v1.1.1

ORRERY_MEMORY.md 自動掛到 Claude

切 env 之後 Claude 看不到之前存的記憶，因為各 env 的 Claude auto-memory 資料夾本來就是獨立的。這版在 orrery create 如果有選 Claude tool，就會自動在那個資料夾裡放一個 symlink 指向 orrery 的共享記憶，這樣跨 env 也能看到同一份。同時也修了 orrery update 的 Homebrew 快取問題 — 現在會先跑一次 brew update 再 upgrade。

v1.1.0

default 改名為 origin

為什麼要改名？因為 default 語意太模糊了：它指的是「orrery 的預設環境」還是「沒被 orrery 動過的系統原狀」？久了連自己都會分不清。改成 origin 一目了然 — 就是「原點」，你還沒進 orrery 之前的那個狀態。既有環境跟設定檔都會自動遷移，不用手動處理。
