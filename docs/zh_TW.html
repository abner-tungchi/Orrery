v2.6.2

解決 AI 多帳號的救星

為 Claude Code、Codex、Gemini CLI 隔離帳號，自由切換工作與個人情境 — 對話歷史無縫接續，不再從頭開始。

開始使用 GitHub
原生安裝（macOS、Linux、WSL）— 推薦
$ curl -fsSL https://offskylab.github.io/Orrery/install.sh | bash
或透過 Homebrew 安裝（macOS）
$ brew install OffskyLab/orrery/orrery
Windows — 透過 WSL
Windows 上的 Claude Code 跑在 WSL 裡。請先以系統管理員身份開啟 PowerShell 啟用 WSL，再到 WSL shell 裡用上面的原生安裝指令安裝 Orrery。
PS> wsl --install
啟用 Shell 整合
$ source ~/.orrery/activate.sh
從 APT 遷移（Linux，v2.3.x 或更早）

如果你之前用 APT（apt install orrery）安裝，跑 orrery update 可能會看到 already the newest version (2.3.x)。APT repo 已不再維護，而且舊版的 update 流程沒有先跑 apt update。只要跑一次原生安裝指令就能完成遷移：

$ curl -fsSL https://offskylab.github.io/Orrery/install.sh | bash

這會移除 APT 管理的舊 binary、安裝新的 orrery-bin，並把之後的 orrery update 切到原生安裝流程。你可以順手清掉已失效的 APT 設定：

$ sudo rm /etc/apt/sources.list.d/orrery.list
$ sudo apt update

無痛切換所需的一切

切換帳號後直接接續對話。Session 資料預設共享 — claude --resume 在切換環境後無縫運作。

Per-shell 生效

orrery use work 只在當前 shell 設定環境變數。其他終端機視窗維持各自的環境，互不影響。

✦

互動式精靈

透過方向鍵與空白鍵選擇工具，引導式建立環境 — 不需要記任何參數。

🔑

自訂環境變數

每個環境可儲存 API 金鑰等變數，orrery use 切換時自動匯出。

⚡

零執行開銷

Swift 原生編譯，啟動瞬間完成。Shell 整合只是一個輕量的 eval 函式，不是背景程式。支援 bash 與 zsh。

核心模型

Orrery 在身分（你以誰的身分登入）與 延續性（你正在做的事情）之間畫出清楚的界線。 切換身分，而不失去 context。

🪐

Environment

隔離身分：帳號、憑證、各工具的設定。透過 orrery use 以 per-shell 啟用 — 其他 terminal 視窗維持自己的環境，互不影響。

💬

Session

代表延續性：對話紀錄與專案 context。預設在所有環境之間共享 — 切換帳號之後 claude --resume 仍然能接續。

🔁

MCP Delegation

在環境之間協作。一個 Claude 實例把任務委派給另一個在不同帳號下執行的實例 — multi-agent workflow，身分邊界清晰。

傳統工具隔離的是相依性（virtualenv）或 runtime 版本（nvm）。 Orrery 把這個概念延伸到 AI 身分、context 與協作。

運作方式

1

安裝 Shell 整合

執行一次 orrery setup。會產生 ~/.orrery/activate.sh 並寫入 rc 檔。再執行 source ~/.orrery/activate.sh 即可在當前 shell 生效。

activate.sh 也會在背景每天檢查是否有新版本，並在下次開啟 shell 時顯示通知。

2

建立環境

執行 orrery create work。互動式精靈讓你選擇工具並設定 session 共享。環境存放在 ~/.orrery/envs/。

3

一行指令切換

orrery use work 將正確的設定目錄與自訂環境變數匯出到當前 session。對話歷史在切換後完整保留。

4

接續上次的對話

切換帳號後，claude --resume 直接接續上一個 session。認證隔離，session 共享 — 兩全其美。

origin 環境

Orrery 預設接管的環境

origin 是 Orrery 首次執行 orrery setup 時接管的環境。 原本的工具設定（~/.claude/、~/.codex/、~/.gemini/） 會被搬進 ~/.orrery/origin/，原路徑改為 symlink — 資料本身完全沒動，只是改住在 Orrery 可以管理與同步的位置。

orrery use origin（或 orrery deactivate）會切回這個預設環境： Orrery 專屬的變數（如 CLAUDE_CONFIG_DIR）會被 unset， 工具走自己原生的路徑 — 而這些路徑透過 symlink 指向 Orrery 管理的 config。 從使用感受上像「離開 Orrery」，但實際上仍在 Orrery 管理的儲存之下。

與一般環境不同，origin 無法被刪除或重新命名，且使用系統原生的 config 路徑。 不過它支援 memory 與 session 設定：可以用 orrery memory isolate / share / storage 控制 memory 存放位置， 也可以在 orrery setup 過程中選擇各工具的 session 是否隔離。 執行 orrery info origin 查看完整狀態。

還原接管。 想讓某個工具完全脫離 Orrery（回到 setup 前的狀態），執行 release：

$ orrery origin status
# 查看目前哪些工具被 Orrery 管理

$ orrery origin release
# 把所有工具還原到原本的位置

$ orrery origin release --claude
# 只釋放 Claude

$ orrery uninstall
# 釋放全部 + 移除 shell 整合

應用情境

💼

工作 / 個人帳號

公司用 Claude Team plan，自己有 Pro 訂閱。不用反覆登入登出，一行指令切換。

$ orrery use work
# → 使用公司 Team plan

$ orrery use personal
# → 使用個人 Pro plan
🏢

多客戶 / 多專案

接案者或顧問同時服務多個客戶，各自有不同的 API key 和憑證。一行指令切換情境。

$ orrery create client-a --tool claude
$ orrery create client-b --tool claude
$ orrery use client-a
💬

切換帳號不中斷對話

用工作帳號跟 Claude 對話到一半？切到個人帳號處理事情，再切回來 — session 還在原地等你。

$ orrery use work
$ claude
# ... 工作中 ...

$ orrery use personal
$ orrery use work
$ claude --resume
# → 接續上次的對話

AI 工具整合

Orrery 透過 MCP 整合 Claude Code、Codex CLI 和 Gemini CLI。跨帳號委派任務、工具間共享記憶、管理 session——全在一個地方。

1

設定一次就好

在專案目錄下執行。Orrery 會註冊為 MCP server 並安裝 slash commands。

$ orrery mcp setup
2

Slash commands

設定完成後，在 Claude Code 裡直接使用：

/orrery:delegate 委派任務給其他帳號
/orrery:delegate 用工作帳號 review 這個 PR 的安全問題
/orrery:sessions 列出 session 並用 index 接續
/orrery:sessions
接著： orrery resume 1
/orrery:phantom 中途切環境 — Claude 自動 --resume 接續對話
啟動時用： orrery run claude
在 Claude 內： /orrery:phantom work
3

或直接用自然語言

Claude 透過 MCP tools 認識你的所有環境。你可以這樣說：

→ 「用工作帳號 review 這個 PR」
→ 「我現在是哪個帳號？」
→ 「列出最近的 session，幫我接續上一個」
4

MCP 工具與共享記憶

所有 AI 工具自動取得以下工具。任一工具儲存的知識，其他工具都能存取。

orrery_memory_read 讀取共享專案記憶
orrery_memory_write 寫入共享專案記憶
orrery_delegate 用其他帳號執行任務
orrery_list 列出所有環境
orrery_sessions 列出當前專案的 session
orrery_current 查看目前啟用的環境
5

外部記憶儲存

Memory 預設儲存在 ~/.orrery。你可以將它重新導向到任何目錄，包括 Obsidian 等 wiki 系統，讓 AI 記憶和你的筆記放在一起。

$ orrery memory storage ~/Documents/my-wiki/orrery
新路徑目前沒有記憶，是否要將目前的記憶複製過去？
▶ 複製過去
不複製（從空白開始）

Fragment 機制與 AI 整合方式不變，AI 直接讀寫你的 wiki 資料夾。

P2P 記憶同步 實驗性功能

跨機器、跨團隊即時同步專案記憶，點對點直連。

🌐

桌機 + 筆電

同一人、兩台電腦。在同一網路下啟動 daemon，Bonjour 自動發現彼此。

# 桌機
$ orrery sync daemon --port 9527

# 筆電（Bonjour 自動發現）
$ orrery sync daemon --port 9528
👥

團隊協作

建立團隊、分享邀請碼。隊友加入後啟動 daemon，所有人的記憶即時同步。

$ orrery sync team create my-team
$ orrery sync team invite --port 9527
# → 分享邀請碼

# 隊友加入：
$ orrery sync team join <code>
$ orrery sync daemon --port 9528
🔒

加密傳輸（mTLS）

敏感環境適用。雙方必須出示同一 CA 簽發的憑證，未授權的節點在握手階段即被拒絕。

$ orrery sync daemon --port 9527 \
  --tls-ca ca.pem \
  --tls-cert node.pem \
  --tls-key node-key.pem
🌍

跨網段

不同辦公室、不同網路。輕量的 rendezvous server 負責節點發現，資料直接在節點間傳輸。

# VPS 上：
$ orrery sync rendezvous --port 9600

# 各節點：
$ orrery sync daemon --port 9527 \
  --rendezvous rv.example.com:9600

僅同步專案記憶 — session 留在本機（因為它綁定機器路徑）。新成員首次連線時透過 manifest reconcile 取得所有現有記憶。記憶變更以無衝突的 fragment 追蹤，由 AI agent 在 session 啟動時整合。

版本歷程

版本說明 — 完整技術細節在 CHANGELOG。

v2.6 最新

Phantom 環境切換：/orrery:phantom <env> 中途切 env

orrery run claude 現在預設啟用 phantom 監督。在 Claude 裡輸入 /orrery:phantom <env> 時，supervisor 會用新環境 + --resume 重啟 Claude，對話無縫接續，不用重開 terminal。要關掉 phantom 走單次模式：orrery run --non-phantom claude。Slash command 同時裝在全域（~/.claude/commands/，給 origin）和專案內（透過 orrery mcp setup）—— project-local 的 slash command 不受 CLAUDE_CONFIG_DIR 影響，所以非 origin 環境也能用。

install.sh 在把 binary 放進 /usr/local/bin 之後會 strip macOS quarantine xattr 並重新 ad-hoc 簽章，修正 curl 安裝後 Gatekeeper SIGKILL（exit 137 / "Killed: 9"）導致 post-install orrery setup 步驟失敗的問題。

v2.4.4

uninstall 移除 binary · thirdparty bundle 修正

orrery uninstall 現在也會刪除 orrery-bin binary，一個指令完整移除。orrery thirdparty 在全新安裝後的 fatal error 已修正：release tarball 和 deb 套件現在會同時打包 orrery_OrreryThirdParty.bundle。

v2.4.3

orrery thirdparty · uninstall rc 殘留修復 · 動態更新通知

新增 orrery thirdparty 指令群組，管理第三方外掛套件：install <id>、uninstall <id>、list、available。套件從 Git 取得（支援本地 vendored cache 離線使用），透過宣告式 manifest 安裝至目前環境的工具設定目錄。--env 為選填，省略時預設使用目前的 active 環境。

orrery uninstall 現在能完整移除 rc 檔中的 lazy-bootstrap stub。舊版只清除 comment 和 source 行，orrery() { … } 的函式本體會殘留。動態更新通知：_check-update 現在從 repo 取得並快取由維護者控制的通知訊息，符合條件時顯示於「有新版本」的訊息旁。

v2.4.1

activate.sh 升級後自動修復 · clone 不再複製帳號資料 · Claude 原生安裝指令

activate.sh 第一行現在帶有版本戳記。每次開新 shell，_orrery_init 會比對戳記版本與安裝的 binary 版本。若不符（例如 brew upgrade 後 post_install 靜默跳過），會立即重新產生並 re-source — shell 不需要任何手動操作就能自我修復。

Clone（orrery create / orrery tools add 複製設定時）不再帶走帳號相關資料。blocklist 從 4 項擴充至 20 項：跳過 cache/、agent-memory/、statsig/、telemetry/、usage-data/、mcp-needs-auth-cache.json、paste-cache/、shell-snapshots/、history.jsonl、file-history/、debug/、downloads/、plans/、tasks/、todos/。保留：settings.json、commands/、skills/、plugins/、agents/、CLAUDE.md、statusline.sh。

Claude 安裝指令改為原生安裝腳本（curl -fsSL https://claude.ai/install.sh | bash）。安裝失敗時現在會顯示手動執行的完整指令，而不是 Swift enum case 名稱。安裝過程中遮蔽錯誤輸出的 alternate-screen buffer 已移除。

v2.4.0

Binary 改名 orrery → orrery-bin · rc 改用 lazy-bootstrap stub

使用者看到的 orrery 指令現在**完全**是 shell function — 真正的 binary 改名 orrery-bin，不再開放給使用者直接呼叫。這徹底解掉一個長期的坑：過去在沒有 source 過 activate.sh 的 shell 裡跑 orrery use work，binary 無法改父 shell 的環境變數，結果靜默失敗。現在這條路直接變成 command not found，明確且容易修復。

orrery setup 改在 rc 檔寫入一個精簡的 lazy-bootstrap stub，取代舊的 source ~/.orrery/activate.sh。shell 啟動成本幾乎歸零 — activate.sh 只在你第一次呼叫 orrery 時才載入。舊的 source …/activate.sh 與 eval "$(orrery setup)" 寫法會自動被遷移。

安裝 / 升級流程（install.sh、Homebrew formula、deb postinst）都會清掉舊的 /usr/local/bin/orrery，確保唯一進入點就是 shell function。新的 tarball 內是 orrery-bin；install.sh 同時也支援舊格式 tarball（內含 orrery），避免過渡期卡死既有 curl 安裝的機器。

MCP 整合改指向 orrery-bin mcp-server（MCP host 以非互動 subprocess 啟動 server，沒辦法走 shell function）。如果你之前有設定過 MCP，請重跑一次 orrery mcp setup。

v2.3.3

原生安裝腳本 · orrery update 改聰明 · setup 安裝後自動執行

macOS / Linux / WSL 的推薦安裝方式改為 curl -fsSL https://offskylab.github.io/Orrery/install.sh | bash。Homebrew 仍然保留作為 macOS 的替代方案。APT repo 已停用 — 維護成本太高。

orrery update 現在更聰明。macOS 上會用 brew list orrery 偵測是否透過 Homebrew 安裝，有的話走 brew upgrade，否則直接重跑 install script。Linux 一律重跑 install script。

orrery setup 在安裝後會自動執行 — curl install script 與 Homebrew formula 的 post_install hook 都會直接呼叫它。一個指令就完成 activate.sh 產生、rc 檔修補、origin 接管。新 shell 自動生效，當前 shell 只要 source ~/.orrery/activate.sh 就好。

安裝文件排版對齊 Claude 的 install 頁面（原生安裝推薦在第一、Homebrew 第二、WSL 註記給 Windows）。origin 區塊也改為 Orrery 預設接管的環境，而非「系統透通」。

v2.3.2

Origin claude 修正 · Linux credential 支援 · clone 精簡 · Claude 改用官方 installer

修正 orrery info origin 與 orrery auth store --env origin 的 Claude 查詢：origin 環境下 CLAUDE_CONFIG_DIR 是 unset，Claude 會用「系統預設」來存 credential — Keychain service 沒有 hash 後綴、.claude.json 放在家目錄根（不是 managed dir 內）。現在兩個指令都按照這個慣例查詢，Claude 的 email / plan / keychain 名稱都能正確顯示。

Linux 上 Claude credential 查詢改讀 {configDir}/.credentials.json（Claude Code 在非 macOS 平台的 JSON 格式），不再誤用 macOS Keychain 路徑。

orrery create --clone 改為只複製真正有用的設定：跳過 cache、agent-memory、telemetry、usage-data、statsig、paste-cache、shell-snapshots、history.jsonl、file-history、debug、downloads、plans、tasks、todos。只有 settings.json、commands/、skills/、plugins/、agents/、CLAUDE.md、statusline.sh 會帶過去。

orrery setup / orrery tools add 安裝 Claude 改用官方 installer（curl -fsSL https://claude.ai/install.sh | bash），取代原本的 npm install -g。安裝失敗時訊息會顯示可手動執行的指令，而不是 enum case 的原始字串。

v2.3.1

orrery auth store — 查看各工具的 credential 儲存位置

新增 orrery auth store，顯示每個工具 credential 實際存放的位置：Claude 顯示 macOS Keychain 的 service 名稱（Claude Code-credentials-{hash}）；Codex 與 Gemini 顯示 credential 檔案路徑（auth.json / oauth_creds.json）。若工具是用 API key 登入，也會一併顯示 masked key（前 4 碼 + ****）。支援 --env <name> 指定環境，以及 --claude / --codex / --gemini 篩選單一工具 — 指定 tool flag 時輸出會切換為純值，方便腳本擷取。

orrery info 在每個工具的 summary 底下也加了 credential-store 子行（Claude 顯示 keychain: …，Codex/Gemini 顯示 file: …），API key 登入時 masked key 也會出現在 summary 裡。

v2.2.4

orrery setup：修正 KILL crash + origin 環境支援 memory/session 設定

所有 FileHandle.write(Data(...))（ObjC API）已全數換成 posix write() syscall，集中到新的 PosixIO.swift。ObjC API 在寫入失敗時會拋出 NSFileHandleOperationException，Swift 無法捕捉，結果是 SIGABRT — 在 iTerm2 裡看起來就是 KILL。posix syscall 遇到失敗只傳回負值，不拋例外。共修改 14 個檔案。

orrery setup 的 session/memory 設定提示現在只在當次 run 新接管的 tool 出現，不再每次 orrery setup 都對所有已管理的 tool 重複顯示。

orrery info origin 現在輸出與一般環境一致的完整結構：Name、Path、Description、Tools（含登入資訊）、Memory Mode、Memory Path、Session Mode、Env Vars。orrery memory isolate / share / storage 現在也支援 origin 環境（設定存在 ~/.orrery/origin/config.json）。

v2.2.3

orrery delegate：修正 pipe-buffer deadlock

在 Claude Code 內部以 Bash tool 呼叫 orrery delegate 時，父層用 pipe 接收輸出，buffer 約 64 KB。長一點的 delegate 任務（code review、多步驟工作）很容易在結束前就超過這個量。舊的 process.standardOutput = FileHandle.standardOutput + waitUntilExit() 寫法導致 buffer 滿後 child 卡在 write()，orrery 卡在 waitUntilExit() — 教科書級的 pipe-buffer deadlock。修法：stdout 與 stderr 改走 Pipe，用 readabilityHandler 在 background queue 上持續 drain，讓 subprocess 存活期間 buffer 永不填滿。

v2.2.2

orrery list：修正 pipe-buffer deadlock、平行化 lookup、Memory 路徑改顯示資料夾

orrery list 在裝了多個 MCP server（figma、notion 等）的機器上會無限卡死 — 那些 OAuth tokens 會被塞進 Claude 的 Keychain credential JSON，整包 size 超過 macOS pipe buffer（約 16 KB，有時更小）。ClaudeKeychain.findPassword 的順序是先呼叫 security、再 waitUntilExit()、最後才讀 pipe — 教科書級的 pipe-buffer deadlock：security 在寫滿的 pipe 上 block，orrery 在等 security 結束，互相等對方。直接跑 security find-generic-password（25 毫秒回傳完整 credential）就能驗證 Keychain 本身沒問題。修法：在 waitUntilExit 之前先把 pipe 讀乾淨。同樣的 pattern 也在 MCPServer.execCommand，現在 stdout 與 stderr 用兩個 background queue 平行 drain（序列讀的話另一支還是會在中途滿掉再 deadlock）。

每個 env 的 Claude/Codex/Gemini 帳號查詢以前是序列的，env 1 的 Keychain 讀取慢，env 2..N 全部跟著等。environmentRows 現在把所有 (env, tool) pair 攤平成一個 work list，用 DispatchQueue.concurrentPerform 派發。worst-case wall time 從 O(N×M×單次) 降到大約 O(單次)。輸出格式與順序不變。

orrery info 與 orrery memory info 以前印的是 .../ORRERY_MEMORY.md — 一個從未真正存在的檔案。v1.1.0 原始設計是寫單一 ORRERY_MEMORY.md 再 symlink 進 Claude 的 auto-memory 目錄；v1.1.2 改成對整個目錄做 symlink，讓所有 auto-memory 寫入直接落在共享可同步路徑，這之後 ORRERY_MEMORY.md 這個檔名就變成 code 一直在追的死重。Memory Path 現在顯示資料夾（例如 ~/.orrery/shared/memory/{projectKey}/）— MEMORY.md 與 fragment 檔真正住的地方。

MCP 工具 orrery_memory_read / orrery_memory_write 現在操作該資料夾內的 MEMORY.md，對齊 Claude auto-memory 慣例 — Codex 與 Gemini（透過 MCP 呼叫這些工具）讀寫的就是 Claude session 啟動時讀的同一份檔案。內部重構：EnvironmentStore.memoryFile() → memoryDir()；memory export 預設輸出檔名改為 MEMORY.md。

v2.2.0

CLI 字串外部化、日文 stub、orrery list 重排

所有 CLI 字串改放在 Sources/OrreryCore/Resources/Localization/<locale>.json，不再硬寫在 Swift 裡。SPM build plugin 在每次 swift build 時讀 JSON、產生 typed accessor 與內嵌的翻譯表 — 完全沒有 runtime resource lookup，所以單檔部署（Homebrew、.deb）照樣 work。CI 還會檢查各 locale 的 key 集合與 placeholder 是否一致，缺一個就 build fail，避免半翻譯版本流到 release。日文（ja.json）已接好，但內容暫時沿用英文，等翻譯完成；runtime 透過 AppLocale.detect() fallback 到英文。

orrery list 改成多行排版：每個環境自成一塊，每個 tool 一行，前面加 ·。一旦 env 多個 tool 或 suffix 變長（email + plan + model），這比舊版單行密集排版好讀很多。當前環境 header 用青色強調，其餘走灰階梯度（email 近白 → plan 中灰 → model 暗）— 資料保持高可讀性，又不會被顏色干擾。輸出 pipe 或被 MCP 讀取時 ANSI 會自動去除。

v2.1.2

Gemini 終於認 orrery 環境，delegate --gemini 也認 API key

gemini-cli 不認 GEMINI_CONFIG_DIR、永遠讀 ~/.gemini/，導致 per-env 隔離一直失敗。現在每個 env 旁邊都會有一個 gemini-home/ 目錄，裡面的 .gemini 會 symlink 回該 env 的 gemini config；orrery use 會 export ORRERY_GEMINI_HOME，shell 端的 gemini() wrapper 用 HOME=$ORRERY_GEMINI_HOME 跑 gemini，落地點就對了。整套 setup 是 idempotent，現有 env 在下一次 orrery use 時會自動補齊。

orrery delegate --gemini 在 API key 模式下原本會失敗 — gemini-cli 的非互動 validator 只看 process.env.GEMINI_API_KEY，不會 fallback 到自己的 Keychain 或加密檔。delegate 現在會先把儲存的 key 撈出來（先試 macOS Keychain，再用跟 gemini-cli 一樣的 scrypt + AES-256-GCM 解 gemini-credentials.json）注入 child process。orrery list 也會讀 settings.json、把 gemini 的 auth type 顯示出來 — gemini(api key) / gemini(vertex) / OAuth email。

v2.1.1

orrery delegate 在非 TTY 呼叫端不再卡 3 秒

orrery delegate 從 script、SSH 沒 pty、或 MCP server 呼叫時，Claude（其他工具也一樣）會印 “no stdin data received in 3s” 警告、然後卡 3 秒才繼續。被 delegate 的工具其實是用 CLI argument 拿 prompt 的，所以 child 的 stdin 改接到 /dev/null，不再繼承呼叫端的 stdin。警告跟 latency 一起消失。

v2.1.0

orrery delete 不帶參數會開多選

測試完一堆 throwaway env 要清掉時，舊版只能 orrery delete <name> 一個一個刪。現在不帶名字直接 orrery delete 會跳 multi-select — 上下鍵移動、空白鍵勾選、Enter 確認，一次刪完。--force 跳過確認；帶名字仍然是原本的單一刪除流程。

同時修了一批：orrery create --tool X 仍會跑該 tool 的 sub-wizard（login 來源、clone 來源、sessions、memory），不再被誤跳過；clone 不再透過 backups/ heal 步驟洩漏來源 identity；self-login 與 clone 會剝掉 identity + onboarding，讓 Claude 重新 prompt；migration 後遺漏的 .claude.json 會從 backup 自動修；migration 留下的 stale symlink 與 origin tool login 也修了。

v2.0.0

改名為 Orrery

為什麼要改名？因為 orbital 作為軟體名稱太常見了，搜尋起來常常撞到別的工具。這版改成 orrery（原意是太陽系機械模型）— 主題上更貼近這個專案「維護多個平行設定宇宙」的精神，名稱也夠冷門。

內容沒有任何功能差異，純粹就是改名：CLI binary orbital → orrery、設定目錄 ~/.orbital/ → ~/.orrery/、環境變數 ORBITAL_* → ORRERY_*、Swift module OrbitalCore → OrreryCore、Homebrew tap OffskyLab/orrery/orrery。

自動搬家：第一次跑 orrery 時，如果偵測到 ~/.orbital/（而且沒有 ~/.orrery/），會自動把目錄整個搬過來，順便重新產生 activate.sh、更新 rc 檔的 source 路徑。舊的 Orbital repo 保留一版 deprecated wrapper — 你現有的 orbital 指令、MCP 設定、shell alias 都還會正常運作，只是會印 deprecation 提示。

v1.1.6

換帳號不用再放棄已經調好的設定

以前想把一個 env 複製出來、換個帳號用會遇到兩難：重新登入麻煩，但整份複製又會連「哪個帳號」都被覆蓋掉，連帶 theme、plugin、已經看過的提示全都一起變了。這版我們把「要用哪個帳號登入」和「要複製誰的設定」拆成兩個獨立的步驟 — 你可以從 A 拿帳號、從 B 拿偏好，.claude.json 會自動幫你合併：身份從登入來源來，其他偏好從 clone 來源來，per-account 快取清掉讓 Claude 自己重抓。

同時每個 tool 都有自己的 login + clone 流程，建立環境時一個一個問「新增並設定？」，不再一口氣把三個工具綁在一起。orrery tools 也拆成 tools add / tools remove；session 共享/隔離改成 per-tool，「Claude 共享、Codex 隔離」這種組合終於行得通；orrery list 跟 orrery info 也會直接顯示每個 tool 登入的 email 和方案。

v1.1.5

Wizard 選完會自己清乾淨

為什麼要動 wizard 的顯示？因為之前每選完一步，那整塊選單都還留在畫面上，跑完全部步驟後 scrollback 滿滿都是中間過程，真正重要的摘要反而被埋住。這版每步選完自動清掉，只留最終結果；create 完後也會直接問你要不要立刻切過去，省得再手動跑一次 orrery use。順手也拿掉了早就不能用的 claude auth login 步驟 — 它根本不認 CLAUDE_CONFIG_DIR，留著只會誤導。

v1.1.4

版本檢查不再卡 shell 啟動

用 Powerlevel10k instant prompt 的朋友反應 shell 一啟動 prompt 就會被 orrery 的版本檢查打斷，原因是我們在 shell init 階段同步做了這件事。這版改成「你下次跑 orrery 指令時順便在背景查一下」，最多 4 小時一次。發現新版時會用黃色訊息出現在每次 orrery 指令前，直到你執行 orrery update 才會消失。

v1.1.3

orrery use 跨 shell 會記得住

為什麼要修這個？因為以前在一個 terminal 切到 work，打開另一個 terminal 卻發現自己又回到了 origin — 其實是 orrery use 切完之後沒把結果寫進 current pointer，新開的 shell 根本讀不到。這版每次 use 成功都會正確寫入，新開的 shell 就會自動回到你上次使用的環境。

v1.1.2

Memory 不用再在 CLAUDE.md 手動寫指示

以前想讓 Claude 讀 orrery 的共享記憶，你得自己去 CLAUDE.md 裡加一段「請去某某路徑讀」的指示，Claude 才會理你 — 很違反「預設就 work」的精神。這版每個專案的 Claude auto-memory 資料夾會自動 symlink 到 orrery 的共享記憶，什麼設定都不用寫，Claude 自己就會讀到；跨 env、跨機器都是同一份。

v1.1.1

ORRERY_MEMORY.md 自動掛到 Claude

切 env 之後 Claude 看不到之前存的記憶，因為各 env 的 Claude auto-memory 資料夾本來就是獨立的。這版在 orrery create 如果有選 Claude tool，就會自動在那個資料夾裡放一個 symlink 指向 orrery 的共享記憶，這樣跨 env 也能看到同一份。同時也修了 orrery update 的 Homebrew 快取問題 — 現在會先跑一次 brew update 再 upgrade。

v1.1.0

default 改名為 origin

為什麼要改名？因為 default 語意太模糊了：它指的是「orrery 的預設環境」還是「沒被 orrery 動過的系統原狀」？久了連自己都會分不清。改成 origin 一目了然 — 就是「原點」，你還沒進 orrery 之前的那個狀態。既有環境跟設定檔都會自動遷移，不用手動處理。
